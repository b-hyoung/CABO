# Commit Hygiene Scorecard (핀 6개 기준)

본 문서는 GitHub에 **고정(Pinned)된 6개 레포지토리**를 대상으로, 협업 지표(PR/리뷰/CI 등)는 제외하고 **본인 커밋 데이터만** 활용해 개발 습관을 구조화하여 보여주기 위한 “커밋 위생(Commit Hygiene)” 분석 기준입니다.

> ⚠️ 이 지표는 코드의 “실력”을 판정하지 않습니다.  
> 커밋 메시지/변경 규모/작업 분할 등 **작업 정리 습관의 신호**를 정량화합니다.

---

## 1) 지표 개요 (추천 7개)

### 1) 메시지 의미도 (Meaningfulness)
- **정의:** “의미 없는 커밋 메시지” 비율
- **예시 패턴:** `update`, `fix`, `test`, `wip`, `temp`, `.`
- **해석:** 비율이 높을수록 위생이 낮음 (의도가 추적 불가)

---

### 2) 메시지 정보량 (Information Density)
- **정의:** 커밋 메시지 길이(문자 수)의 분포
  - 중앙값/평균
  - “너무 짧은 메시지” 비율(예: 8~12자 이하)
- **해석:** 지나치게 짧은 메시지가 많으면 변경 의도 파악이 어려움

---

### 3) 구조화 비율 (Structure)
- **정의:** 구조화된 메시지 패턴의 비율
  - Conventional Commits 비율: `feat:`, `fix:`, `refactor:`, `docs:` …
  - 또는 `[scope]`, `(#123)` 같은 구조 패턴
- **해석:** 구조화 비율이 높을수록 변경 기록의 검색/분류가 쉬움

---

### 4) 작업 분할 성향 (Commit Size Hygiene)
- **정의:** 커밋당 변경량(added + deleted) 분포
  - 중앙값/평균
  - “거대 커밋” 비율(예: 500+, 1000+ 라인)
- **해석:** 거대 커밋이 많으면 변경 리뷰/추적이 어려워짐 (작업 분할 약함)

---

### 5) 변경 범위 관리 (Blast Radius)
- **정의:** 커밋당 변경 파일 수 분포
  - 중앙값/평균
  - “다수 파일 커밋” 비율(예: 30개+)
- **해석:** 변경 범위가 큰 커밋이 잦으면 영향 범위 파악이 어려움

---

### 6) 리듬 (Consistency-lite)
> 위생이라기보다 **작업 습관 신호**에 가까워 별도 섹션으로 분리 가능
- **정의:**
  - 활동 주 커버리지: 최근 90일 중 커밋이 있었던 주 비율
  - 주간 변동성: 주별 커밋 표준편차 / 평균
- **해석:** 커버리지/변동성으로 작업의 “간헐성 vs 지속성”을 간접 확인

---

### 7) 되돌림/급한 수정 신호 (Optional)
- **정의:**
  - `revert` 커밋 비율
  - `hotfix`, `quick fix` 패턴 빈도
- **주의:** 너무 과하면 품질이라기보다 운영/릴리즈 압박의 신호일 수 있어 **보조 지표**로만 사용

---

## 2) 결과 표현 방식 (단정 금지)

“100점” 같은 단일 점수는 오해를 만들기 쉬우므로 **3단계 등급 + 근거 수치**로 표시합니다.

- **Clean**
  - 의미 없는 메시지 비율 낮음
  - 거대 커밋 비율 낮음
- **Mixed**
  - 일부 지표는 양호하나 특정 항목 개선 여지 있음
- **Messy**
  - 메시지 의미도가 낮거나(의미 없는 메시지 비율 높음)
  - 거대 커밋/광범위 변경 비율이 높음

> 등급은 “판정”이 아니라 **관찰 가능한 신호의 요약**입니다.

---

## 3) 핀 6개 기준 추천 UI/리포트 구성

### A. 계정 전체 요약 (최근 90일)
- 커밋 위생 요약(핵심 지표 3~5개)
- 작업 리듬(커버리지/변동성) 요약

### B. 핀 6개 리포별 카드 (각 리포 최근 90일)
- 메시지 의미도 / 정보량 / 구조화
- 커밋 크기 / 변경 파일 수
- (옵션) 리듬 및 되돌림 신호

### C. “자세히 보기”(상위 1~2개 리포)
- 대표 커밋 예시 2~3개(중립적으로)
  - 좋은 예: 의도/범위가 명확한 메시지
  - 개선 예: 너무 짧거나 의미 없는 메시지
- ⚠️ 비꼬는 톤 금지, “사실 기반 예시”로만

---

## 4) 구현 팁 (속도/정확도)

### 메시지 위생(1~3)
- **GitHub API만으로 충분**
  - 커밋 메시지 텍스트 기반 분석
  - 구조 패턴 매칭(정규식)

### 커밋 크기/파일 수(4~5)
- API만으로 하면 호출량이 커질 수 있음  
- **핀 6개만 shallow clone 후 numstat로 계산**이 더 빠를 수 있음
  - 예: `git log --since="90 days ago" --numstat`

---

## 5) 이 접근이 개인 프로젝트에 적합한 이유
- PR/리뷰/CI가 없어도 작동함
- 프로젝트가 여러 개여도 비교 가능(정규화 가능)
- “판정”이 아니라 **행동 기록(커밋 로그)** 기반이라 반발이 적음

---

## 6) 확장(선택)
- 기존에 가진 활동 지표(주간 평균/표준편차/시간대 비율)와 결합하여
  - “리듬” 섹션을 강화하거나
  - 위생과 습관을 분리해 더 설득력 있는 프로필 구성 가능
